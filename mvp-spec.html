Enablement Diagnostic MVP Implementation Spec

This spec translates the public launch workflow into concrete build requirements. It defines the MVP scope, frontend fields, payload structure, roadmap-generation outputs, email flow, feedback loop, and analytics requirements so the product can move from concept to build without ambiguity.

MVP goalPublic diagnostic that generates a tailored 30-60-90 PDF roadmap and emails it to the recipient.
Primary promiseImmediate on-screen diagnosis + delivered PDF roadmap.
Support loopBuilt-in feedback and use-case submission path.
Later phaseSame engine reused inside Docebo after public version is stable.
Section 1

MVP scope

The MVP is intentionally narrow: one public-facing diagnostic, one submission flow, one roadmap-generation flow, one email-delivery flow, and one feedback flow. Docebo is explicitly deferred.

🎯

In scope for MVP

  • Public HTML diagnostic page
  • Recipient email field
  • On-screen diagnosis output
  • Webhook or backend submission endpoint
  • Tailored 30-60-90 roadmap generation
  • PDF creation
  • Email delivery
  • Feedback / use-case submission button
  • Submission logging for analytics

Out of scope for MVP

  • Docebo embedding / widget implementation
  • Authentication / login gating
  • Advanced admin dashboard UI
  • Multi-language localization
  • Enterprise CRM or ticketing integration
  • Complex user accounts or saved histories

Section 4

30-60-90 roadmap PDF spec

The roadmap should not just restate the diagnosis. It should translate the diagnosis into sequenced action over 30, 60, and 90 days.

🗂️

Required PDF sections

  • Title page / brand header
  • Diagnostic summary
  • Primary issue and risk level
  • Why this is / is not primarily a training issue
  • 30-day priorities
  • 60-day priorities
  • 90-day priorities
  • Owner guidance
  • Evidence to watch
  • Recommended next actions

Roadmap generation rules

  • 30-day section = diagnosis validation + immediate structural fixes
  • 60-day section = system changes, support structures, and workflow refinements
  • 90-day section = reinforcement, measurement, governance, and optimization
  • Roadmap content should vary based on primary diagnosis type
  • Each stage should be concise, action-oriented, and non-generic

Example 30-60-90 logic by diagnosis

Primary diagnosis 30 days 60 days 90 days
Workflow design Map current state, identify friction points, validate breakpoints Redesign sequence, simplify handoffs, update support tools Reinforce new workflow and measure execution consistency
Governance Clarify owners, approvals, and escalation paths Document and publish governance standards Review governance adherence and reduce exceptions
Support structure Define missing support points and manager roles Launch reinforcement tools and manager-facing supports Measure support usage and improve weak areas
Visibility Identify blind spots and reporting gaps Create or refine dashboard and signal views Use signals to drive earlier intervention and better decisions
Section 5

Email-delivery spec

The email flow is part of the product promise. It should be clear, branded, and reliable.

✉️

Trigger

User clicks the CTA to receive the roadmap after completing the diagnostic and providing a valid email address.

Email contents

Short branded message, attached PDF roadmap, optional link back to the diagnostic, and optional feedback link.

Delivery status

System should store whether email send succeeded, failed, or needs retry.

{
  "emailTemplate": {
    "subject": "Your Enablement Diagnostic 30-60-90 Roadmap",
    "bodySections": [
      "thank_you",
      "short_summary",
      "pdf_attached",
      "feedback_link"
    ]
  },
  "deliveryLog": {
    "submissionId": "uuid",
    "emailSent": true,
    "emailProviderMessageId": "provider-message-id",
    "pdfGenerated": true,
    "sentAt": "ISO_TIMESTAMP"
  }
}
Section 6

Feedback and use-case submission spec

The feedback button should be treated as part of the product loop, not an afterthought. It is how the tool becomes better and how public interest can be surfaced into future opportunities.

💬

Feedback button behavior

  • Visible on the results screen
  • Visible in the follow-up email
  • Opens feedback form or modal
  • Can optionally prefill user email

Feedback fields

  • Email
  • Role / organization
  • How they used the tool
  • What was most helpful
  • What was unclear
  • Improvement suggestions
  • Whether they want follow-up
Section 7

Analytics and logging requirements

The MVP should capture enough signal to evaluate tool usage, delivery success, and the most common diagnostic themes without building a full analytics product on day one.

📊

Minimum event logging

  • Diagnostic page opened
  • Diagnostic started
  • Diagnostic completed
  • Email submitted
  • Roadmap generation attempted
  • PDF generation succeeded / failed
  • Email send succeeded / failed
  • Feedback form opened
  • Feedback form submitted

Minimum reporting dimensions

  • Submission timestamp
  • Primary diagnosis type
  • Severity
  • Audience
  • Context
  • Risk level
  • Roadmap sent or not
  • Feedback submitted or not
